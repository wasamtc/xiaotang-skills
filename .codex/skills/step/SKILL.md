# 分步可确认

## 启用方式

只有在用户显式点名这个 skill 时才启用，例如：

- `step`
- `$step`
- “使用 step skill”
- “按 step 方式处理”

以下情况都不要自动启用：

- 用户只是说“分步修改”
- 用户只是说“先给计划”
- 用户只是说“每次改完等我确认”
- 用户只是表达想慢一点推进，但没有显式点名 `step`

## 工作流程

严格按下面顺序执行：

1. 先只给修改计划，不要改任何文件。
2. 每次只修改 1 个文件，且只做 1 类改动。
3. 改完后立即停止，并输出：
   - 改了什么
   - 为什么改
   - 影响范围
4. 等用户回复“继续”后，再做下一步。
5. 如果涉及多个函数，先从最小的一处开始。

## 响应模板

第一次响应只给计划：

```text
Step 1 计划
- 目标文件:
- 改动类型:
- 为什么改:
- 预期影响:

等待确认。
```

每一步完成后使用：

```text
已完成本步
- 改了什么:
- 为什么改:
- 影响范围:

回复“继续”后进入下一步。
```

## 约束

- 不要把多个不相关改动合并到同一步。
- 如果任务涉及多个文件，拆成多个确认轮次。
- 如果用户要求先看 diff，就先给 diff，不要直接改。
- 如果用户说“停止”“暂停”“重做”或要求调整方案，不要继续下一步。
- 每一步都应足够小，便于快速审阅和确认。
